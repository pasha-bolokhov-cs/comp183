Lab 4 hints

Multiplication is an expensive (i.e. time-consuming) operation (only division is more expensive among ALU operations).

The "obvious" and easy way to multiply is to do repeated addition, like this python program:

def multiply (n1, n2):
    result = 0
    while n1 > 0:
        result += n2           
        n1 -= 1
    return result

if __name__ == '__main__':
    number1 = input ("Enter a number ")
    while number1 != 0:         # can't just Enter to input ()
        number2 = input ("Enter a number ")
        print number1, "*", number2, "=", multiply (number1, number2)
        number1 = input ("Enter a number ")

But this can clearly be inefficient: if I multiply 3 by 1200 I go around the loop 3 times, but if I multiply 1200 by 3, I go around it 1200 times...

Of course, the function multiply could always test its parameters and switch them if they are the "wrong way" round, but if we are multiplying, say, 200 by 200....

But there is a way (known since virtually the first computers were built) to do it in O(n) time where n is the length of the registers.

Basically, we're talking long multiplication.

Consider this 110 * 101. You should have learned in school to do it like this:

	110
	101
	---
	110
       0000
      11000
      -----
      11110

Note the cunning I show in choosing those numbers: in decimal one hundred and ten times one hundred and one is eleven thousand, one hundred and ten. In binary six times five is thirty....

We save a little time by not adding in the zero partial products (row two) and our algorithm looks like this:

  • result = 0
  • while n1 != 0
    • if last bit of n1 is a 1: add n2 to result
    • shift n1 1 bit to the right
    • shift n2 1 bit to the left

Your multiply procedure needs to get the two numbers from the stack (as discussed in the notes) and then get (also from the stack) the address of (i.e. a pointer to) the result, which is where you put the answer. Putting the answer in the stack will not work.

Definitions for Bit0 through Bit15 are already included in my driver so you can test (use the TEST instruction!) n1 (in whichever register you've put it) against Bit15.

FYI: the above algorithm only works for unsigned numbers. And some machines - including the Intel x86 chips - have two multiply instructions, one for signed numbers one for unsigned, although on the Intel chips the unsigned multiply is called IMUL (Integer MULtiply) for some unfathomable reason.

For a more complete algorithm, but a little tricky for those who aren't mathematically inclined, there is  Booth's algorithm which works for both signed and unsigned numbers.  I may discuss this in class depending on where we're at.
